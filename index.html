User access review · Workiva Wdesk

How it works, and what we build to make it work

The automated UAR program walks through eleven stages, from an automatic trigger to a locked audit record. Open any stage below to see the mechanism, the data it touches, and the specific assets we need to build.

The lifecycle

Eleven stages. Click to expand the build detail for each.
01
Cycle trigger
Open a review cycle automatically, on a fixed cadence.
▼
What happens
A new UAR cycle opens automatically on its configured cadence. Opening the cycle starts every SLA clock and kicks off ingestion, triggering the connection process each system supports (automated API feed, scheduled job, or manual upload).
How (technology)
A Workiva Chains scheduled (cron) trigger node. Cadence is set per system, quarterly for critical apps, semi-annual for lower-risk ones.
Inputs
A cycle-configuration table: system list, cadence, due dates, and SLA tier per system.
Assets to build
Cycle scheduler ChainCycle_Config table
Validations & checks
Skip systems not yet in scope; prevent a second cycle opening for a system that already has one running.
Output
A "cycle open" event that launches the per-system ingestion process downstream.
02
Data ingestion
Pull the raw entitlement and identity feeds for the cycle.
▼
What happens
Each source system feeds the cycle through whichever connection it supports. The connection method varies by system, but all paths land in one write-locked staging table, the raw base population for the cycle.
How each system connects
APIAutomated feed
System exposes an API. A Chain pulls the extract automatically when the cycle opens.
NO APIScheduled job
No API available. A scheduled job on the system pushes the extract (for example over secured SFTP) on a timer.
MANUALManual upload
No automated path. The system admin uploads a secured, encrypted extract into the cycle.
→
One write-locked staging table
All three paths land in a single staging table in Wdata: the raw, untransformed base population for the cycle.
How (technology)
A Workiva Chains pipeline per system, with the connection step matched to what the system supports: pull via API, receive a scheduled push, or accept a manual upload. Each writes into the shared staging table.
Data sources (4 to confirm)
App_Entitlement_Extract per systemUser_Role_MappingHR_Employee_Supervisor_MappingHR_Employee_email_useraccount_mapping
Assets to build
Ingestion Chains (API / scheduled / manual)Unified write-locked staging table
Validations & checks
Row counts compared against the prior cycle; checksum on manual uploads; reject null primary keys.
Output
One raw, write-locked staging table in Wdata, holding the base population from every source.
03
HR & role merge OPTIONAL STEP
Join the feeds into one unified population row. Depends on HR-to-account linkage.
▼
What happens
Where HR-to-account linkage is available, the raw feeds are joined into one human-readable row per user, role, and system, enriched with the person's HR identity, manager, status, and title. This enrichment is an optional step, see the note below.
Optional, depends on existing HR-to-account linkage
This step only works if there is a reliable key between the HR system and the user records in the target system (email or Employee ID). Where that linkage already exists, the merge runs automatically and the step is turned on. Where it does not, the mapping has to be built first, and the step can stay off until it is.
Small user base
Links can be mapped manually as a one-time setup.
Larger user base
Establishing the mapping is a larger effort for management to scope. The step stays off until it is in place.
How (technology)
Wdata SQL JOIN queries across the staging tables.
Merge & keys
Account ↔ HR identity on email / Employee ID (HR_Employee_email_useraccount_mapping)
Account ↔ role on user ID (User_Role_Mapping)
Employee ↔ manager on employee ID (HR_Employee_Supervisor_Mapping)
Assets to build
Unified_Population query
Validations & checks
Flag accounts with no HR match (orphans) and employees with no manager, both are exceptions that need resolving before review.
Output
An enriched population: user, status, title, role, access level, manager, last login. When the step is off, the population carries access and role data without the HR enrichment.
04
Role mapping & reviewer resolution
Attach role meaning and the right reviewer, then the coordinator confirms the assignments.
▼
What happens
Every row is matched to a role definition and the reviewer responsible for it. The system checks for roles not in the mapping table and holds them until an approver is named.
Coordinator confirmation at setup
Before any sheets are generated, the review coordinator manually reviews the role → reviewer assignments and confirms the people named at setup are still the right approvers. This sign-off is a gate: reviewer sheets are only created once the coordinator confirms.
How (technology)
A Wdata lookup against a Role_Reviewer_Mapping table, plus conditional logic for new roles and role-change dates.
The mapping table holds
Role → reviewer / owner, role description, sensitivity, and assignment model (A / B / C).
Key checks
New-role check — role not in mapping → flag and require a new approver before proceeding.
Role last-changed date — pulled per role so reviewers see what changed.
Assets to build
Role_Reviewer_Mapping tableNew_Role exception reportCoordinator confirmation gate
Output
Every row tagged with a reviewer, role description, sensitivity, and role-change date, confirmed by the coordinator and ready for sheet generation.
05
Reviewer sheet assembly
Build one fully-contexted sheet per reviewer, after coordinator confirmation.
▼
What happens
Each reviewer gets their own sheet, scoped to only their rows. Every row carries the context needed to make a decision.
How (technology)
Once the review coordinator confirms the assignments, a Wdata query split by reviewer generates the per-reviewer Wdesk review sheets.
What each reviewer row carries
Current employee status · current job title (per HR) · current role and access level · description of what the role can do · when the role was last changed · a link to browse all role definitions.
Assets to build
Reviewer-sheet generatorRole Definitions referenceCompleteness & accuracy reconciliation
Completeness & accuracy
A reconciliation confirms the row count across all reviewer sheets equals the confirmed population, so no one is lost in the split. Every in-scope row maps to exactly one reviewer, no gaps and no duplicates, and new-role rows stay held until an approver is assigned.
Output
One ready-to-review sheet per reviewer. See the reviewer screen further down the page.
06
Line certification
Reviewers decide each line: Approve, Revoke, or Escalate.
▼
What happens
Reviewers work their sheet row by row and record a decision with a reason. The context they need, including last cycle's decision, sits on the same row.
How (technology)
Controlled picklists and data validation in the Wdesk review sheet (Approved · Revoke · Escalate · Pending).
Assets to build
Decision columns + validation
Validations & checks
A rationale is mandatory for Revoke and Escalate; the escalation target must resolve to a valid, active user.
Output
A decision and justification on every row.
07
Completeness lock
Stop an incomplete review from being submitted.
▼
What happens
The sheet cannot be submitted while any row is undecided. A validation gate on the submit action blocks submission until every row has a decision.
How (technology)
A Wdesk Workflow validation gate on the submit action.
Assets to build
Workflow submit-gate rule
Validations & checks
Submit is blocked while any row is Pending or missing a required justification.
Output
A sheet with a decision on every row before it moves on.
08
Coordinator roll-up & escalations
Aggregate every reviewer sheet; triage and close escalations.
▼
What happens
Every per-reviewer sheet rolls up into one review coordinator sheet. The coordinator tracks overall progress, and escalations are reviewed and closed here.
How (technology)
Cross-workbook cell links and Wdata aggregation into a single coordinator workbook.
Assets to build
Review Coordinator sheetEscalation queue
Validations & checks
All escalations must be resolved and closed before certification; stalled escalations are flagged to the coordinator.
Output
One consolidated, escalation-clean view ready for the system owner. See the coordinator view further down the page.
09
System certification
The system owner applies one binding sign-off.
▼
What happens
With reviews complete and escalations closed, the system owner reviews the consolidated assessment report and applies a single, legally binding sign-off for their system.
Consolidated user-access assessment report
A core audit deliverable. For every user, the report consolidates each access decision made across all reviewers and systems into one record: roles held, access level, the final disposition (approved, revoked, or escalated), the rationale, and who certified it. A follow-up view flags users with open items, revocations not yet executed or unresolved escalations, building on the coordinator roll-up (step 8) and remediation (step 10).
How (technology)
The Workiva Certifications module, with a binding e-signature.
Assets to build
Certification node (per system)Consolidated user-access assessment report
Validations & checks
Sign-off only unlocks when the coordinator sheet is complete and escalations are closed. The owner can send rows back for rework.
Output
A signed, locked certification record.
10
Closed-loop remediation
Execute revocations and prove they happened.
▼
What happens
Every Revoke becomes a tracked remediation item. The remediation lead executes the removal in the target system and uploads system-generated proof, closing the loop.
How (technology)
Revoke rows post to the exceptions / remediation ledger, which downstream de-provisioning polls. Removal happens outside Workiva and is evidenced back in.
Assets to build
Exceptions / remediation ledger
Validations & checks
Each revocation needs system-generated proof of removal, not a screenshot. An SLA timer runs per item.
Output
A closed-loop ledger of removals, each with evidence.
11
Audit lock
Freeze the cycle into immutable evidence.
▼
What happens
The cycle closes and everything converts to read-only, append-only evidence, exportable for audit.
How (technology)
Certification completion combined with a read-only permission switch, on top of the platform's un-deletable audit trail.
Assets to build
Evidence export pack
Validations & checks
No edits possible post-lock; the audit trail records who did what, when.
Output
Locked, exportable evidence: population snapshot, certified review log, signed certificate, and the remediation ledger.
